# Stack adapter — Markdown fallback (universal)

When the framework isn't directly supported, emit content as markdown files with structured frontmatter. The user wires them into their templating layer once; the skill keeps producing markdown for every phase after.

## File layout (universal)

```
content/seo/
├── alternatives/
│   ├── hootsuite.md
│   ├── buffer.md
│   └── ...
├── for/
│   ├── agencies.md
│   ├── social-listening.md
│   └── ...
├── compare/
│   ├── buffer-vs-hootsuite.md
│   └── ...
└── playbooks/
    ├── b2b-social-media-strategy.mdx
    └── ...
```

## How the user wires this into their template

After the skill writes its first markdown file, hand off with this note:

> The skill writes content as markdown to `content/seo/`. You'll need to wire your framework to read these files and render them. Common patterns:
>
> - **Hugo:** drop the files in `content/seo/alternatives/`, use a custom layout per content type
> - **Jekyll:** use collections (`_alternatives/`, etc.) configured in `_config.yml`
> - **Eleventy:** use a `_data/` directory or collection front-matter
> - **Gatsby:** source-filesystem plugin + `createPages` in `gatsby-node.js`
> - **VitePress/Docusaurus:** treat as MDX pages
> - **WordPress:** use a custom theme that parses these or import as posts
>
> Once wired, every subsequent phase just adds another file in the same directory. The skill handles the data; you handle the rendering.

## Sitemap (universal)

Most SSG frameworks generate sitemaps automatically once content is in the right collection. If yours doesn't:

```
content/seo/sitemap-data.yaml         # generated by skill, alphabetical
```

Then have your build step emit `/sitemap.xml` from this file. Example shape:

```yaml
last_modified: 2026-MM-DD
urls:
  - loc: https://example.com/alternatives/hootsuite
    changefreq: monthly
    priority: 0.8
  - loc: https://example.com/for/agencies
    changefreq: monthly
    priority: 0.8
  # ...
```

## File-write checklist for any phase (markdown mode)

- [ ] New markdown file at the right `content/seo/<pattern>/<slug>.md` path
- [ ] Frontmatter validates against the pattern schema
- [ ] Sibling files updated to add the new slug under `related_internal_links` (≥2 spots)
- [ ] `sitemap-data.yaml` updated if used
- [ ] User's build step ran successfully (rebuild + verify HTML output)
- [ ] HTML output: `grep -c '<h1' build-output/alternatives/<slug>/index.html` — must be >0
- [ ] `docs/seo-sprint.md` tracker row updated in same commit
